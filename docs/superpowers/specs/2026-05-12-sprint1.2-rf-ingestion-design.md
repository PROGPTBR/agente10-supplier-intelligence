**Status:** approved 2026-05-12

## 0. Contexto

Spec do Sprint 1.1 deferiu para 1.2 a popularização da tabela `empresas` com dados da Receita Federal e a criação dos índices `GIN(cnaes_secundarios)` e `GIST(geom)`.

### Decisões fixadas no brainstorming (2026-05-12)

Brainstorming explorou se APIs públicas eliminariam a necessidade do bulk RF. Conclusão: **não**. APIs grátis (BrasilAPI, ReceitaWS, Conecta gov.br) expõem apenas lookup por CNPJ individual; nenhuma oferece busca por CNAE+UF retornando listas. Alternativas pagas (Cortex, Casa dos Dados) custam R$200-2k/mês recorrente. Alternativa moribunda (Nuvem Fiscal) sai do ar em 31/07/2026.

Caminho escolhido — **bulk RF local via tooling comunitário existente**:

- Usar **`rictom/cnpj-sqlite`** AS-IS como script vendored em `/scripts` (último release v0.7, mar/2026, já com fix para layout RF jan/2026)
- Escrever **transform Python** que lê o SQLite intermediário e faz UPSERT em batches no Postgres `empresas`, denormalizando empresa+estabelecimento+simples
- Carregar apenas **situação cadastral = ATIVA** (~35M empresas; baixadas/inaptas/suspensas dropadas)
- **Geocoding deferido** para Sprint 1.2.5. `geom` fica NULL; GIST não é criado neste sprint
- **Discovery helper minimalista** análogo ao `top_k_cnaes()` do Sprint 1.3, com golden test de validação ponta-a-ponta
- **Filiais incluídas** (cada estabelecimento tem CNPJ próprio e pode ser fornecedor real)

## 1. Objetivo

Popular `empresas` com ~35M registros ATIVOS da Receita Federal e expor um helper de discovery por CNAE+UF/município, criando os índices estruturais que essa query precisa.

### Definition of done

- `make ingest-rf` baixa e parseia o dump RF mais recente via `rictom/cnpj-sqlite`, produzindo `backend/data/cnpj.db` (~30GB, gitignored)
- `make load-empresas` lê o SQLite, filtra ATIVAS, denormaliza e UPSERT em `empresas` — idempotente
- Após carga: `SELECT COUNT(*) FROM empresas` retorna ~30-35M (margem para variação mensal RF)
- Migration `0006_empresas_gin.py` cria `GIN(cnaes_secundarios)`, reversível
- Helper `find_empresas_by_cnae(cnae, uf?, municipio?, limit=25)` retorna candidatas tipadas (`EmpresaCandidate`)
- Golden test `test_golden_recall` valida que CNPJs conhecidos (ex.: Petrobras `33000167000101` em CNAE `1921700`) aparecem no top-25 do helper
- Unit + integration-shape tests rodam em CI sem precisar do bulk; golden test só roda se base populada (skip-hook `rf_ingested`)
- `make lint-backend` zero warnings

### Fora de escopo (explícito)

- **Geocoding** (`geom` permanece NULL) — Sprint 1.2.5
- **Filtro por distância** (`ST_DWithin`) — depende de geom — Sprint 1.2.5
- **GIST(geom) index** — sem dados, defer
- **Ranking composto** (porte, capital_social, score CNAE match) — quando Estágio 3 existir
- **Update automatizado** (cron, diff detection) — sprint dedicada futura
- **`socios` table** — NUNCA (LGPD; spec proíbe explicitamente desde o briefing técnico §9.3)
- **Fuzzy match razão social** (pg_trgm) — quando consumidor concreto pedir
- **`spend_cluster` → empresas pipeline** — Estágio 3 sprint
- **Migração RF → ranqueamento por concentração (HHI)** — vestigial em modo catálogo (memory)

## 4. Tooling externo — `rictom/cnpj-sqlite`

Tratado como ferramenta vendored, **não como dependência Python**. Não vai em `pyproject.toml`.

`scripts/run_cnpj_sqlite.sh` faz:

```bash
#!/usr/bin/env bash
# Wrapper para rictom/cnpj-sqlite.
# Idempotente: clona se ausente, pulls se já existe.
set -euo pipefail

CACHE_DIR="${CACHE_DIR:-/tmp/cnpj-sqlite-cache}"
TARGET_DB="$(dirname "$0")/../data/cnpj.db"

if [ ! -d "$CACHE_DIR" ]; then
    git clone --depth 1 https://github.com/rictom/cnpj-sqlite.git "$CACHE_DIR"
else
    git -C "$CACHE_DIR" pull --ff-only
fi

cd "$CACHE_DIR"
python -m venv .venv 2>/dev/null || true
.venv/bin/pip install -q -r requirements.txt
.venv/bin/python cnpj-sqlite.py  # ← nome conforme repo deles; verificar na execução

mv cnpj.db "$TARGET_DB"
echo "OK — SQLite em $TARGET_DB"
```

**Verificações na execução real (NÃO assumir):**

- Nome exato do entry script do rictom (pode ser `dados_cnpj.py`, `cnpj_full.py`, etc.)
- Lista exata de `requirements.txt` do rictom
- Caminho onde o SQLite final é gerado

O implementador do sprint precisa rodar o repo upstream uma vez localmente para confirmar.

### 5.1 Tabelas RF relevantes (no SQLite intermediário)

`rictom/cnpj-sqlite` produz aproximadamente estas tabelas (nomenclatura pode variar — checar empiricamente):

```
empresa(cnpj_basico PK, razao_social, natureza_juridica, qualificacao_responsavel,
        capital_social, porte_empresa, ente_federativo)
estabelecimento(cnpj_basico, cnpj_ordem, cnpj_dv,
                identificador_matriz_filial, nome_fantasia, situacao_cadastral,
                data_situacao_cadastral, motivo_situacao_cadastral,
                cidade_exterior, pais, data_inicio_atividades,
                cnae_fiscal_principal, cnae_fiscal_secundaria,  -- CSV string
                tipo_logradouro, logradouro, numero, complemento, bairro,
                cep, uf, municipio,  -- código IBGE municipio
                ddd_1, telefone_1, ddd_2, telefone_2, ddd_fax, fax,
                correio_eletronico, situacao_especial, data_situacao_especial)
simples(cnpj_basico, opcao_simples, data_opcao_simples,
        data_exclusao_simples, opcao_mei, data_opcao_mei, data_exclusao_mei)
municipio(codigo PK, descricao)
natureza_juridica(codigo PK, descricao)
qualificacao_socio, motivo, pais, cnae  -- lookups
```

### 5.2 Query principal do transform

```sql
SELECT
    e.cnpj_basico || est.cnpj_ordem || est.cnpj_dv  AS cnpj,
    e.razao_social,
    NULLIF(est.nome_fantasia, '')                   AS nome_fantasia,
    est.cnae_fiscal_principal                       AS cnae_primario,
    est.cnae_fiscal_secundaria                      AS cnaes_sec_csv,
    'ATIVA'                                         AS situacao_cadastral,
    est.data_inicio_atividades                      AS data_abertura,
    e.porte_empresa,
    e.capital_social,
    NULL                                            AS faixa_funcionarios,
    e.natureza_juridica,
    est.uf,
    m.descricao                                     AS municipio_nome,
    est.cep,
    -- endereço concatenado
    trim(coalesce(est.tipo_logradouro,'')||' '||coalesce(est.logradouro,'')||
         ', '||coalesce(est.numero,'')||' '||coalesce(est.complemento,'')||
         ' - '||coalesce(est.bairro,''))            AS endereco,
    est.ddd_1 || est.telefone_1                     AS telefone,
    est.correio_eletronico                          AS email
FROM estabelecimento est
JOIN empresa e USING (cnpj_basico)
LEFT JOIN municipio m ON m.codigo = est.municipio
WHERE est.situacao_cadastral = '02'  -- 02 = ATIVA conforme layout RF
ORDER BY e.cnpj_basico, est.cnpj_ordem
```

`02` é o código da situação cadastral ATIVA conforme tabela `motivo_situacao_cadastral` da RF. O transform DEVE validar empiricamente este código contra o `motivo` lookup do SQLite produzido pelo rictom (defensive — RF já mudou códigos no passado).

## 6. Migration 0006 — `GIN(cnaes_secundarios)`

```python
# backend/alembic/versions/0006_empresas_gin.py
"""empresas: GIN index em cnaes_secundarios

Revision ID: 0006
Revises: 0005
Create Date: 2026-05-12
"""

from alembic import op

revision = "0006"
down_revision = "0005"
branch_labels = None
depends_on = None


def upgrade() -> None:
    op.execute(
        """
        CREATE INDEX IF NOT EXISTS empresas_cnaes_secundarios_gin
        ON empresas USING GIN (cnaes_secundarios);
        """
    )
    op.execute("ANALYZE empresas;")


def downgrade() -> None:
    op.execute("DROP INDEX IF EXISTS empresas_cnaes_secundarios_gin;")
```

**Nota sobre `CONCURRENTLY`:** spec 1.1 sugeriu `CREATE INDEX CONCURRENTLY`. Alembic não suporta CONCURRENTLY em transação (e migrações rodam em transação por padrão). Sprint 1.2 usa CREATE INDEX bloqueante porque:

1. Migrations rodam pré-produção (build/CI) — sem tráfego concorrente
2. Para 35M rows o index criado vai bloquear ~5-15min — aceitável em janela de manutenção
3. Em produção real depois, com tráfego, mudar para CONCURRENTLY via DDL fora do Alembic

## 7. Discovery helper — `src/agente10/empresas/discovery.py`

```python
"""Discovery de empresas candidatas a fornecedoras, por CNAE + filtros opcionais."""

from datetime import date
from pydantic import BaseModel
from sqlalchemy import text
from sqlalchemy.ext.asyncio import AsyncSession


class EmpresaCandidate(BaseModel):
    cnpj: str
    razao_social: str
    nome_fantasia: str | None
    cnae_primario: str
    primary_match: bool  # True se cnae casou em cnae_primario; False se em secundarios
    uf: str | None
    municipio: str | None
    data_abertura: date | None


_QUERY = text(
    """
    SELECT
        cnpj,
        razao_social,
        nome_fantasia,
        cnae_primario,
        (cnae_primario = :cnae) AS primary_match,
        uf,
        municipio,
        data_abertura
    FROM empresas
    WHERE (cnae_primario = :cnae OR :cnae = ANY(cnaes_secundarios))
      AND situacao_cadastral = 'ATIVA'
      AND (CAST(:uf AS text) IS NULL OR uf = :uf)
      AND (CAST(:municipio AS text) IS NULL OR municipio = :municipio)
    ORDER BY primary_match DESC, data_abertura ASC NULLS LAST
    LIMIT :limit
    """
)


async def find_empresas_by_cnae(
    db: AsyncSession,
    cnae: str,
    uf: str | None = None,
    municipio: str | None = None,
    limit: int = 25,
) -> list[EmpresaCandidate]:
    """Top-N empresas ATIVAS cujo CNAE primário OU secundário contém ``cnae``.

    Ordenação: matches em primário antes de matches em secundário; dentro de
    cada grupo, mais antigas primeiro (proxy de estabilidade).
    """
    result = await db.execute(
        _QUERY,
        {"cnae": cnae, "uf": uf, "municipio": municipio, "limit": limit},
    )
    return [
        EmpresaCandidate(
            cnpj=row.cnpj,
            razao_social=row.razao_social,
            nome_fantasia=row.nome_fantasia,
            cnae_primario=row.cnae_primario,
            primary_match=bool(row.primary_match),
            uf=row.uf,
            municipio=row.municipio,
            data_abertura=row.data_abertura,
        )
        for row in result.all()
    ]
```

```python
# tests/integration/test_discovery.py
pytestmark = pytest.mark.integration


@pytest.mark.asyncio
async def test_shape_with_synthetic_fixture(db_session):
    """Shape test usando 100 empresas sintéticas — sem dump RF real."""
    await db_session.execute(
        text(Path("tests/fixtures/cnpj_synthetic.sql").read_text())
    )
    candidates = await find_empresas_by_cnae(db_session, "4744001", limit=10)
    assert all(isinstance(c, EmpresaCandidate) for c in candidates)
    assert all(len(c.cnpj) == 14 for c in candidates)
    # primary_match=True deve vir antes de False
    primary_then_secondary = [c.primary_match for c in candidates]
    assert primary_then_secondary == sorted(primary_then_secondary, reverse=True)


@pytest.mark.rf_ingested
@pytest.mark.asyncio
async def test_golden_recall(db_session):
    """Golden — só roda se base populada com >1M empresas."""
    rows = list(csv.DictReader(open(GOLDEN_CSV)))
    misses = []
    for r in rows:
        candidates = await find_empresas_by_cnae(
            db_session, r["cnae"], uf=r["uf"], limit=25
        )
        cnpjs = [c.cnpj for c in candidates]
        if r["cnpj_esperado"] not in cnpjs:
            misses.append((r["descricao"], r["cnpj_esperado"], cnpjs[:5]))
    recall = 1 - len(misses) / len(rows)
    assert recall >= 0.80, f"recall@25 = {recall:.2f}; misses: {misses}"
```

### 8.4 Skip-hook `rf_ingested`

Adicionar em `conftest.py`:

```python
def pytest_collection_modifyitems(config, items):
    # ... voyage hook do Sprint 1.3 já presente ...

    # Skip 'rf_ingested' marker se base vazia
    skip_rf = pytest.mark.skip(reason="empresas não populada (<1M); rode `make ingest-rf` + `make load-empresas`")

            if "rf_ingested" in item.keywords:
```

```toml
markers = [

    "rf_ingested: requires empresas populated via `make load-empresas` (skip otherwise)",
```

## 10. Riscos e mitigações

| Risco | Mitigação |
|---|---|
| Layout RF muda novamente | `rictom/cnpj-sqlite` upstream traz fix; nosso transform tem validação defensiva do código `situacao_cadastral='02'` |
| `rictom/cnpj-sqlite` é abandonado | Substituível por `aphonsoar/Receita_Federal_*` em 1-2 dias de trabalho — só o wrapper bash muda |
| 35M UPSERTs lentos | Batches de 100k em transação por chunk; `COPY ... FROM STDIN` como otimização se >2h |
| Disco do dev/CI estourar | Documentado em docker-compose; CI nunca executa o bulk |
| Encoding (RF usa Latin-1?) | rictom lida internamente; nosso transform abre SQLite, não os CSVs brutos |
| CNPJs alfanuméricos no dump | Validar com `validar_cnpj_dv()` do Sprint 1.1 num sample; transform aceita VARCHAR(14) que cabe alfa |
| Golden test instável (empresa baixa CNPJ usado) | Picks são corporações grandes (Petrobras, Itaú); revisar empiricamente antes de cada release maior |
| `ANY(VARCHAR(7)[])` performance sem GIN | Migration 0006 cria GIN explicitamente; medir `EXPLAIN ANALYZE` durante implementação |
