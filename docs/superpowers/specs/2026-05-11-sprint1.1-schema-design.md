## 0. Mudança de escopo do produto (importante)

Antes deste sprint, o briefing assumia que o input do Agente 10 seria **spend transacional** (linhas com data, valor, quantidade, fornecedor). O cliente piloto enviou amostras de planilhas que mostram um padrão diferente: **catálogo de categorias de materiais** com classificações CNAE manualmente atribuídas/revisadas. Sem valores nem datas.

**Implicações:**

| | Briefing original | Cenário catálogo (atual) |
|---|---|---|
| Input | linhas transacionais | catálogo (agrupamento + descrição + cnae sugerido + notas) |
| Estágio 1 | classificar do zero, por linha | **cluster + validar** classificação existente, por cluster |
| Estágio 2 (HHI) | ativo | vestigial (sem valor) |
| Estágio 3 | fornecedores por CNAE priorizado | fornecedores por CNAE validado |
| Estágio 4 (curator com Serasa) | ativo | vestigial (sem priorização) |

**Decisão tomada (modo dual):** o produto suporta **modo catálogo** (lançamento) e **modo transacional** (futuro próximo) sem refactor. Schema preparado para ambos. Aplicação ramifica via `spend_uploads.modo`.

## 1. Objetivo

Lançar a fundação de dados:

1. Schema completo (10 tabelas: 4 globais + 6 tenant-specific) suportando modo catálogo e modo transacional
2. RLS policies em 5 das 6 tabelas tenant-specific (todas exceto `tenants`), com testes de isolamento
3. Índices estruturais (PKs, FKs, btree de filtro). Índices custosos (ivfflat, GIN, GIST, pg_trgm) ficam para sprints que os usem
4. Validador CNPJ alfanumérico (vigência jul/2026) + utilitários + testes
5. Helper `tenant_context` para multi-tenancy via SET LOCAL

### Definition of done

- `make migrate` aplica três migrations (0002, 0003, 0004) sem erro a partir do schema atual (só `tenants` mínimo do Sprint 0)
- `psql -c "\dt"` mostra 10 tabelas + `alembic_version` + `spatial_ref_sys` (PostGIS)
- `validar_cnpj_dv` aceita CNPJs alfanuméricos válidos (jul/2026) e legacy só-numéricos; rejeita inválidos
- `make test` cobre validador (≥95% coverage, property-based via Hypothesis) e RLS isolation (2 tenants em 2-3 tabelas representativas)
- `make lint` zero warnings
- Spec coverage de todas as 10 tabelas + RLS + validador
- Testes RLS rodam em CI (postgres service) sem mocks

### Fora de escopo

- Carregar dados em qualquer tabela (Sprint 1.2 popula `empresas`; 1.3 popula `cnae_taxonomy`; 1.4 popula `empresa_signals.em_ceis/cnep`)
- Geocoding (Sprint 1.2)
- Embeddings Voyage-3 (Sprint 1.3)
- Job scheduling (Sprint 1.5)
- Endpoints REST consumindo as tabelas (Sprint 2+)
- Frontend (Sprint 2+)

## 4. RLS policies

### 4.1 Padrão aplicado às 6 tabelas tenant-specific

(spend_uploads, spend_linhas, spend_clusters, concentracao_categorias, supplier_shortlists)

```sql
ALTER TABLE <tabela> ENABLE ROW LEVEL SECURITY;
ALTER TABLE <tabela> FORCE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation_select ON <tabela>
  FOR SELECT
  USING (tenant_id = current_setting('app.current_tenant_id', true)::UUID);

CREATE POLICY tenant_isolation_modify ON <tabela>
  FOR ALL
  USING (tenant_id = current_setting('app.current_tenant_id', true)::UUID)
  WITH CHECK (tenant_id = current_setting('app.current_tenant_id', true)::UUID);
```

`FORCE ROW LEVEL SECURITY` é necessário: o user da app é dono das tabelas; sem FORCE bypassa RLS. Migrations DDL (que não fazem DML cross-tenant) continuam funcionando porque RLS só afeta DML.

`current_setting('app.current_tenant_id', true)::UUID` — segundo arg `true` retorna NULL se var não setada (em vez de exception). Defesa em profundidade: queries sem `tenant_context` retornam 0 linhas.

### 4.2 `tenants` sem RLS

A tabela `tenants` em si não recebe RLS — auth/JWT verifica que o `tenant_id` requisitado pertence ao usuário antes de chamar `tenant_context`. Adicionar RLS aqui seria circular.

## 5. `tenant_context` helper

```python
# backend/src/agente10/core/tenancy.py
from contextlib import asynccontextmanager
from collections.abc import AsyncIterator
from uuid import UUID

from sqlalchemy import text
from sqlalchemy.ext.asyncio import AsyncSession


@asynccontextmanager
async def tenant_context(
    session: AsyncSession, tenant_id: UUID | str
) -> AsyncIterator[AsyncSession]:
    """Set the current tenant for RLS via SET LOCAL.

    The setting is transaction-scoped — automatically reverted on commit/rollback.
    Must be called inside an active transaction (begin/begin_nested).
    """
    await session.execute(
        text("SET LOCAL app.current_tenant_id = :tid"),
        {"tid": str(tenant_id)},
    )
    yield session
```

**Uso:**

```python
async with session.begin():
    async with tenant_context(session, tenant_a_id):
        # all reads/writes scoped to tenant_a
        ...
# commit auto-resets app.current_tenant_id (SET LOCAL semantic)
```

### 7.4 Testes (`backend/tests/test_cnpj.py`)

Casos obrigatórios:
- **Smoke:** ≥5 CNPJs legacy válidos conhecidos (ex: CNPJs reais públicos de empresas grandes)
- **Smoke:** ≥3 CNPJs claramente inválidos (DV trocado, comprimento errado, chars não-permitidos)
- **Formato:** `limpar_cnpj("11.222.333/0001-44")` retorna `"11222333000144"`
- **Property-based via Hypothesis:** gera 1000 CNPJs via `gerar_cnpj_alfanum_valido()`, todos devem validar
- **Property-based:** gera strings aleatórias de 14 chars [A-Z0-9], maioria deve invalidar (probabilidade de DV acertar é 1/121 ≈ 0.83%)
- **Edge cases:** string vazia, 13 chars, 15 chars, chars não-alfanuméricos (`@#$`), lowercase (`abcdef`...)
- **Alfa puro:** valida pelo menos 1 CNPJ alfanumérico exemplo da especificação RF (a RF tem exemplos no Manual de Orientação do CNPJ Alfanumérico)

Cobertura mínima: 95% em `utils/cnpj.py`.

Dependência nova: `hypothesis` em dev deps de pyproject.toml.

## 8. Testes RLS (`backend/tests/test_tenancy.py`)

### 8.1 Setup

- Marker `@pytest.mark.integration` para distinguir de smoke tests (que não tocam DB)
- Fixture `db_session` — engine real ao Postgres do docker-compose (CI usa o service)
- Fixture `two_tenants` — cria UUIDs `A` e `B`, garante limpeza ao final

### 8.2 Cenários

**Tabelas cobertas:** `spend_uploads`, `spend_linhas`, `spend_clusters` (3 representativas das 6).

Para cada tabela:
1. Sem `tenant_context`: `SELECT * FROM <tabela>` retorna 0 linhas (mesmo que tenha dados de outros tenants)
2. `tenant_context(A)` + INSERT de linha para tenant A: sucesso
3. `tenant_context(B)` + INSERT de linha para tenant B: sucesso
4. `tenant_context(A)` + SELECT: retorna apenas a linha de A
5. `tenant_context(A)` + INSERT com `tenant_id=B`: bloqueado por `WITH CHECK` (raise `IntegrityError` ou similar)
6. `tenant_context(A)` + UPDATE de linha de B: 0 linhas afetadas

Não testar as outras 3 tabelas (`concentracao_categorias`, `supplier_shortlists`): RLS é o mesmo padrão, custo/benefício baixo.

### 8.3 Limpeza

Cada teste usa `session.begin()` + rollback ao final (ou TRUNCATE no fixture teardown se rollback não der conta de algum cenário com commits intermediários).

## 10. Trade-offs registrados

### 10.1 `spend_linhas.cnae` denormalizado

**Decisão:** denormaliza (a linha guarda o `cnae` do cluster).
**Razão:** queries comuns ("todas as linhas com CNAE X") evitam JOIN. Em modo transacional onde não há cluster, o `cnae` é a única fonte.
**Risco:** drift cluster↔linha se app não mantiver consistência. Mitigação: teste de invariância (`linha.cnae == cluster.cnae` para toda linha com `cluster_id IS NOT NULL`).

### 10.2 `FORCE ROW LEVEL SECURITY`

**Decisão:** ON em todas as 6 tabelas tenant-specific.
**Razão:** o user da app é dono das tabelas; sem FORCE ele bypassaria RLS, anulando a proteção.
**Custo:** mínimo — DDL (migrations) não é afetada.

### 10.3 Hypothesis como dependência

**Decisão:** adicionar `hypothesis>=6.0` em dev deps.
**Razão:** property-based testing pega edge cases que parametrize tradicional não pega. Especialmente útil para verificar invariâncias de algoritmos numéricos (módulo 11).
**Alternativa rejeitada:** só parametrize. Cobertura mais frágil, e gerador de CNPJs válidos já vai ser necessário em fixtures de outros testes (`spend_uploads` precisa de um cnpj_fornecedor válido).

### 10.4 `tenants` sem RLS

**Decisão:** RLS apenas nas tabelas que referenciam `tenant_id`. A própria `tenants` fica sem.
**Razão:** auth/JWT verifica que o tenant_id requisitado pertence ao usuário antes de `tenant_context`. RLS em `tenants` seria circular (precisa LER tenants pra saber qual tenant_id setar).
**Alternativa rejeitada:** RLS em tenants baseado em `user_id`. Mais robusta, mas adiciona complexidade desnecessária no MVP onde não há ainda tabela `users`.

### 10.5 CNAE armazenado como 7 dígitos numéricos canônicos

**Decisão:** `VARCHAR(7)` sem `.` ou `/`. Conversão display é da apresentação.
**Razão:** consistência interna; JOINs e índices simples; formato display varia (alguns usam `XX.XX-X/XX`, outros `XXXX-X/XX`).
**Custo:** parsers de input precisam normalizar — utility helper compartilhado.
